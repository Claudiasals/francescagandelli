Guida per principianti: deploy del backend su Render.com

Questa guida spiega passo passo come pubblicare online il server Node.js (Express) del progetto, dopo aver già messo il frontend su Netlify. È pensata per chi non ha mai fatto il deploy di un backend.

1. Concetti base: cosa stai facendo

Frontend e backend sono due programmi diversi

  • Frontend (React su Netlify): sono i file che il browser scarica (HTML, JavaScript, CSS). Netlify li serve come un sito “statico”. Non esegue il tuo codice Node.
  • Backend (Express su Render): è un programma che gira su un computer remoto (un server) sempre acceso (o quasi). Risponde alle richieste tipo “dammi le foto della galleria”, “effettua login”, “salva questa immagine”.

Finché il backend resta solo sul tuo PC (http://localhost:5000), solo tu puoi usarlo. Deploy significa: installare e avviare quel programma su una macchina di un fornitore (qui Render), così ottiene un indirizzo pubblico tipo https://nome-servizio.onrender.com.

Perché Render

Render è un servizio che:

  • prende il codice dal tuo GitHub;
  • esegue npm install e avvia il server con npm start;
  • ti dà HTTPS e un URL stabile.

Esistono alternative (Railway, Fly.io, VPS): la logica è simile (codice + variabili d’ambiente + avvio).

Monorepo: cartella server

Il repository contiene più cartelle. Il backend sta in server/. Su Render devi dire Root Directory = server, così i comandi npm install e npm start girano dentro quella cartella, dove c’è il package.json del backend.

2. Cos’è Nodemailer

Nodemailer è una libreria Node.js per inviare email dal server (non è Gmail né un sito: è codice che gira sul backend).

Nel progetto viene usata nella rotta del form Contatti: quando un visitatore compila il modulo, il backend riceve nome, email e messaggio e manda un’email tramite SMTP (di solito Gmail o altro provider) verso l’indirizzo configurato.

  • Non è un servizio di posta: è il “ponte” nel codice che parla con un server SMTP usando utente e password (o password per app) che metti nelle variabili d’ambiente (EMAIL_USER, EMAIL_PASS).
  • Senza Nodemailer nel package.json, il server potrebbe andare in errore all’avvio quando carica il file che importa Nodemailer.

Per questo nel progetto sono stati aggiunti lo script npm start (node src/server.js) e la dipendenza nodemailer, così Render può avviare il server in produzione e il form può inviare mail.

3. Cosa serve prima di iniziare

  • Account GitHub con il repository aggiornato (stesso repo del frontend).
  • Account Render (gratuito possibile, con limiti).
  • Account MongoDB Atlas (tier gratuito ok per iniziare) con stringa di connessione.
  • Account Cloudinary (caricamento immagini): nome cloud, API key, API secret.
  • Per il form contatti: coppia email SMTP (es. Gmail + password per le app, non la password normale dell’account Google).

4. MongoDB Atlas (database in cloud)

Il backend non salva i dati “dentro Render”: usa MongoDB. In cloud, Atlas ospita il database.

Passi essenziali

  1. Crea un cluster (se non ce l’hai già).
  2. Crea un utente database con username e password (salvali in posto sicuro).
  3. Network Access: consenti le connessioni da Internet verso il cluster. Per iniziare è comune Allow access from anywhere (0.0.0.0/0). La sicurezza dipende anche da password forte e da MONGO_URI segreta su Render.
  4. Copia la connection string (URI). Sostituisci <password> con la password dell’utente DB (se ha caratteri speciali, può servire l’URL encoding). La userai come MONGO_URI su Render.

Se Atlas blocca gli IP da cui si connette Render, il server non parte o va in errore: controlla sempre Network Access.

5. Render.com: creare il Web Service

  1. Vai su dashboard.render.com e accedi.
  2. New + → Web Service.
  3. Collega il repository GitHub (autorizza Render se richiesto) e scegli il repo del progetto.
  4. Compila i campi principali (tabella sotto).
  5. Piano Free: utile per prove; il servizio può andare in sleep. La prima richiesta dopo lo sleep può richiedere decine di secondi.
  6. Environment: aggiungi le variabili della sezione 6 (prima o subito dopo; puoi rifare Manual Deploy dopo modifiche).
  7. Clicca Create Web Service e attendi i log. Se tutto va bene, aprendo la root nel browser vedrai un messaggio tipo che il server è OK.

Sono coppie nome = valore che non vanno nel codice pubblico su GitHub. Render le passa al processo Node all’avvio.

7. CORS spiegato semplice

Il browser, per sicurezza, blocca molte richieste da un sito (es. Netlify) verso un altro dominio (es. Render), a meno che il server dica: “accetto richieste da questo dominio”. Questo meccanismo si chiama CORS.

Nel progetto il server accetta http://localhost:5173 e http://localhost:4173 (sviluppo Vite) più tutti gli URL in ALLOWED_ORIGINS.

Devi inserire l’URL esatto del sito Netlify (di solito https://..., senza slash finale), ad esempio:

https://francescagandelli-photographer.netlify.app

Se sbagli o ometti questa variabile, in console vedrai errori tipo “blocked by CORS policy”.

8. Collegare il frontend (Netlify) al backend

Il frontend spesso punta ancora a http://localhost:5000. In produzione non funziona: il PC del visitatore non ha il tuo server locale.

Le chiamate API devono andare all’URL Render, es. https://TUO-SERVIZIO.onrender.com. Le rotte iniziano con /api/..., quindi la base tipica è:

https://TUO-SERVIZIO.onrender.com/api

Passi consigliati:

  1. Prova nel browser: https://TUO-SERVIZIO.onrender.com/ → risposta di testo dal server.
  2. Aggiorna il frontend con un’unica base URL (meglio variabile VITE_API_URL su Netlify).
  3. Rifai il deploy su Netlify dopo le modifiche.

Finché il frontend punta a localhost, il sito Netlify può aprirsi ma login, gallerie e form non useranno il backend reale.

9. Checklist finale e problemi comuni

Checklist

  • Atlas: cluster, utente DB, Network Access (es. 0.0.0.0/0 per iniziare).
  • Render: Root server, Build npm install, Start npm start.
  • Tutte le variabili d’ambiente su Render.
  • ALLOWED_ORIGINS = URL Netlify esatto.
  • Test URL root Render nel browser.
  • Frontend con URL API di produzione + nuovo deploy Netlify.

Modifiche previste nel codice backend (riferimento)

  • "start": "node src/server.js" in server/package.json.
  • Dipendenza nodemailer.
  • CORS con ALLOWED_ORIGINS (lista separata da virgole) oltre ai localhost.
